Day 1 ‚Äì Modern JavaScript Essentials (SDET-Centric)

üîç What You're Really Learning

Beyond syntax, you're learning how JavaScript's execution model affects test stability. Automation code is often debug-heavy, and misunderstanding references vs values leads to mutated shared state across tests, flaky behavior, and difficult-to-trace bugs.

üí• The #1 JavaScript Pitfall in Automation:

Shared mutable state causes the most subtle, hard-to-diagnose test failures. When test A modifies an object that test B also uses, you get unpredictable results that vary by test execution order.

üìö Deep Dive: Execution Context & Closures

Understanding closures isn't academic‚Äîit's essential for creating isolated test fixtures. A closure lets you encapsulate state that's private to a test helper, preventing accidental sharing between tests. 

// ‚ùå DANGEROUS: Shared state across tests
let requestCount = 0;

function makeRequest() {
  requestCount++;
  console.log(`Request ${requestCount}`);
  // If tests run in parallel, requestCount gets corrupted
}

// ‚úÖ SAFE: Encapsulated state with closure
function createRequestCounter() {
  let requestCount = 0; // Private to this closure
  
  return function makeRequest() {
    requestCount++;
    console.log(`Request ${requestCount}`);
    // Each test gets its own counter instance
  };
}

// In test setup:
const counterA = createRequestCounter(); // Has its own count
const counterB = createRequestCounter(); // Independent count

‚öôÔ∏è Hands-On: Test Configuration Builder

Build a configuration utility that prevents the most common shared state bug in test frameworks: 

// Common bug pattern in test utilities
const defaultConfig = { 
  retries: 2, 
  baseUrl: "https://api.example.com",
  timeout: 30000 
};

// ‚ùå WRONG: Mutates shared default
function createConfig(overrides) {
  // This modifies defaultConfig for ALL future calls!
  return Object.assign(defaultConfig, overrides);
}

// ‚úÖ CORRECT: Creates new object each time
function createConfig(overrides = {}) {
  return { 
    ...defaultConfig, // Spread operator creates shallow copy
    ...overrides      // Overrides applied on top
  };
}

// ‚úÖ EVEN BETTER: Deep clone for nested objects
function createConfig(overrides = {}) {
  const config = JSON.parse(JSON.stringify(defaultConfig));
  return { ...config, ...overrides };
}

Day 2 ‚Äì TypeScript for Reliable Test Automation

TypeScript isn't just "JavaScript with types"‚Äîit's a compile-time test suite that catches bugs before tests even run. For SDETs, TypeScript's real power is in creating self-documenting, error-resistant interfaces for test frameworks.

üéØ TypeScript's Killer Feature for SDETs: Discriminated Unions

Discriminated unions let you model different test outcomes in a type-safe way, eliminating whole categories of assertion bugs: 

// Without discriminated unions - error prone
interface TestResult {
  passed: boolean;
  error?: string;
  duration?: number;
  // What if passed=true but error exists?
}

// With discriminated unions - compiler prevents invalid states
type TestResult = 
  | { type: 'success'; duration: number; details: string }
  | { type: 'failure'; error: string; stackTrace?: string }
  | { type: 'skipped'; reason: string };

// TypeScript ensures you handle all cases
function handleResult(result: TestResult) {
  switch (result.type) {
    case 'success':
      console.log(`‚úì ${result.duration}ms: ${result.details}`);
      break;
    case 'failure':
      console.error(`‚úó ${result.error}`);
      break;
    case 'skipped':
      console.log(`‚ö† Skipped: ${result.reason}`);
      break;
    // No default needed - compiler ensures exhaustive handling
  }
}

üîß Generic Helpers for Test Utilities

Generics let you write flexible utilities that maintain type safety across different test data structures: 

// Generic retry utility with typed errors
export async function retry<T>(
  operation: () => Promise<T>,
  options: {
    retries: number;
    delayMs: number;
    shouldRetry?: (error: unknown) => boolean;
    onRetry?: (attempt: number, error: unknown) => void;
  }
): Promise<T> {
  for (let attempt = 1; attempt <= options.retries; attempt++) {
    try {
      return await operation();
    } catch (error) {
      if (attempt === options.retries) {
        throw error; // Final attempt failed
      }
      
      if (options.shouldRetry && !options.shouldRetry(error)) {
        throw error; // Non-retriable error
      }
      
      if (options.onRetry) {
        options.onRetry(attempt, error);
      }
      
      await new Promise(resolve => setTimeout(resolve, options.delayMs));
    }
  }
  
  // TypeScript knows this is unreachable, but we need it for compilation
  throw new Error('Max retries exceeded');
}

Day 6 ‚Äì Dependency Injection for Testable Services

üéØ Why DI Matters for SDETs

Dependency Injection (DI) is the #1 technique for making code testable. It lets you:

  • Replace real dependencies with mocks during testing
  • Isolate units of code for focused testing
  • Configure different behaviors for different test scenarios
  • Reduce coupling between test components

üîß Three DI Patterns for Automation

// 1. CONSTRUCTOR INJECTION (Most common and recommended)
interface HttpClient {
  get(url: string): Promise<Response>;
}

class ApiService {
  // Dependency is injected at construction time
  constructor(private httpClient: HttpClient) {}
  
  async getUser(id: string) {
    return this.httpClient.get(`/users/${id}`);
  }
}

// Usage in production
const realService = new ApiService(new RealHttpClient());

// Usage in tests
const mockClient = { get: jest.fn() };
const testService = new ApiService(mockClient);

// 2. SETTER INJECTION (When dependencies change at runtime)
class ConfigurableService {
  private httpClient?: HttpClient;
  
  setHttpClient(client: HttpClient) {
    this.httpClient = client;
  }
}

// 3. INTERFACE INJECTION (Most flexible, most complex)
interface HttpClientUser {
  useHttpClient(client: HttpClient): void;
}

class FlexibleService implements HttpClientUser {
  private httpClient?: HttpClient;
  
  useHttpClient(client: HttpClient) {
    this.httpClient = client;
  }
}

export interface IHttpClient {
  get<T>(url: string, options?: RequestOptions): Promise<T>;
  post<T>(url: string, data: unknown, options?: RequestOptions): Promise<T>;
  put<T>(url: string, data: unknown, options?: RequestOptions): Promise<T>;
  delete<T>(url: string, options?: RequestOptions): Promise<T>;
}

  headers?: Record<string, string>;
  timeout?: number;
  retries?: number;
}

  constructor(

    private defaultOptions: RequestOptions = {}
  ) {}

    } catch (error) {

        // Implement retry logic here
      }
      throw error;
    }
  }

Day 7 ‚Äì Composition Over Inheritance in Test Design

‚öñÔ∏è The Inheritance vs Composition Decision

Inheritance creates rigid hierarchies; composition creates flexible systems. For test frameworks, composition almost always wins:

‚ö†Ô∏è The Fragile Base Class Problem:

When test base classes change, all derived test classes break. This causes widespread test failures unrelated to the actual product code being tested.

üé≠ Strategy Pattern for Pluggable Behaviors

Use the Strategy pattern when you need interchangeable algorithms (e.g., different authentication methods): 

// Authentication strategies
interface AuthStrategy {
  authenticate(request: Request): Promise<Request>;
}

class BearerTokenAuth implements AuthStrategy {
  constructor(private token: string) {}
  
  async authenticate(request: Request): Promise<Request> {
    request.headers.set('Authorization', `Bearer ${this.token}`);
    return request;
  }
}

class ApiKeyAuth implements AuthStrategy {
  constructor(private apiKey: string) {}
  
  async authenticate(request: Request): Promise<Request> {
    request.headers.set('X-API-Key', this.apiKey);
    return request;
  }
}

  constructor(private clientId: string, private clientSecret: string) {}
  
  async authenticate(request: Request): Promise<Request> {
    const token = await this.getAccessToken();
    request.headers.set('Authorization', `Bearer ${token}`);
    return request;
  }
  
  private async getAccessToken(): Promise<string> {
    // Implementation to get OAuth2 token
  }
}

// Composable API client
class AuthenticatedApiClient {
  constructor(
    private httpClient: IHttpClient,
    private authStrategy: AuthStrategy
  ) {}

export async function withTimeout<T>(
  promise: Promise<T>,
  timeoutMs: number,
  cleanup?: () => void
): Promise<T> {
  let timeoutId: NodeJS.Timeout;
  
  const timeoutPromise = new Promise<never>((_, reject) => {
    timeoutId = setTimeout(() => {
      cleanup?.();
      reject(new TimeoutError(
        `Operation timed out after ${timeoutMs}ms`,
        timeoutMs
      ));
    }, timeoutMs);
  });
  
  try {
    const result = await Promise.race([promise, timeoutPromise]);
    clearTimeout(timeoutId);
    return result;
  } catch (error) {
    clearTimeout(timeoutId);
    throw error;
  }
}

// Usage in tests

  );

// Cancellable operations
export class CancellableOperation<T> {
  private isCancelled = false;
  private cancelHandlers: (() => void)[] = [];
  
  async run(
    operation: () => Promise<T>,
    onCancel?: () => void
  ): Promise<T> {
    if (onCancel) {
      this.cancelHandlers.push(onCancel);
    }
    
    if (this.isCancelled) {
      throw new Error('Operation was cancelled');
    }
    
    try {
      return await operation();
    } finally {
      // Cleanup
    }
  }
  
  cancel(): void {
    if (this.isCancelled) return;
    
    this.isCancelled = true;
    this.cancelHandlers.forEach(handler => {
      try {
        handler();
      } catch (error) {
        console.error('Error in cancel handler:', error);
      }
    });
    this.cancelHandlers = [];
  }
}

    """Analyze structured test logs for insights."""

    test_logs = [log for log in logs if log.get('event') in {
        'test_started', 'test_passed', 'test_failed', 'test_skipped'
    }]
    
    analysis = {
        'total_tests': len([l for l in test_logs if l['event'] == 'test_started']),
        'passed': len([l for l in test_logs if l['event'] == 'test_passed']),
        'failed': len([l for l in test_logs if l['event'] == 'test_failed']),
        'skipped': len([l for l in test_logs if l['event'] == 'test_skipped']),
        'success_rate': 0,
        'common_errors': Counter(),
        'slowest_tests': [],
        'flaky_tests': []
    }
    
    # Calculate success rate
    if analysis['total_tests'] > 0:
        analysis['success_rate'] = (
            analysis['passed'] / analysis['total_tests'] * 100
        )
    
    # Find common errors in failed tests
    failed_logs = [log for log in test_logs if log['event'] == 'test_failed']
    for log in failed_logs:
        error_type = log.get('error_type', 'unknown')
        analysis['common_errors'][error_type] += 1
    
    # Find slowest tests
    duration_logs = [
        log for log in test_logs 
        if log['event'] in {'test_passed', 'test_failed'} 
        and 'duration_ms' in log
    ]
    analysis['slowest_tests'] = sorted(
        duration_logs, 
        key=lambda x: x['duration_ms'], 
        reverse=True
    )[:10]
    
    return analysis

Question: Design a generic retry utility for failed HTTP calls that will be used inside a Playwright-based test framework. What parameters should it accept, and how would you test it?

üéØ Core Design Parameters:

The retry utility should accept:

  1. operation: Async function to retry (returns Promise)
  2. maxRetries: Maximum number of retry attempts
  3. initialDelayMs: Base delay before first retry
  4. backoffFactor: Multiplier for exponential backoff (e.g., 2 for doubling)
  5. shouldRetry: Predicate function to determine if error is retriable
  6. onRetry: Callback for logging/metrics on each retry attempt
  7. timeoutMs: Overall timeout for the entire operation
üèó Implementation Architecture: 
class RetryableError extends Error {
  constructor(message: string, public readonly retryable: boolean = true) {
    super(message);
  }
}

async function retry<T>(
  operation: () => Promise<T>,
  options: {
    maxRetries: number;
    initialDelayMs: number;
    backoffFactor?: number;
    shouldRetry?: (error: unknown, attempt: number) => boolean;
    onRetry?: (error: unknown, attempt: number, delayMs: number) => void;
    timeoutMs?: number;
  }
): Promise<T> {
  let lastError: unknown;
  
  for (let attempt = 0; attempt <= options.maxRetries; attempt++) {
    try {
      if (options.timeoutMs) {
        return await withTimeout(operation(), options.timeoutMs);
      }
      return await operation();
    } catch (error) {
      lastError = error;
      
      // Check if we should stop retrying
      const isLastAttempt = attempt === options.maxRetries;
      const shouldNotRetry = options.shouldRetry && !options.shouldRetry(error, attempt);
      
      if (isLastAttempt || shouldNotRetry) {
        break;
      }
      
      // Calculate delay with exponential backoff
      const delayMs = options.initialDelayMs * 
        Math.pow(options.backoffFactor || 2, attempt);
      
      // Notify observer
      if (options.onRetry) {
        options.onRetry(error, attempt + 1, delayMs);
      }
      
      // Wait before retry
      await new Promise(resolve => setTimeout(resolve, delayMs));
    }
  }
  
  throw lastError;
}
üß™ Testing Strategy:

Test using a combination of strategies:

  1. Unit tests with mocked time: Use Jest's fake timers to test delays without waiting
  2. Failure simulation: Create a function that fails N times then succeeds
  3. Error classification: Test with retriable vs non-retriable errors
  4. Boundary testing: Test with 0 retries, 1 retry, many retries
  5. Timeout testing: Ensure overall timeout is respected
  6. Concurrency testing: Ensure multiple retry operations don't interfere

enum RecoveryAction {
  RETRY_LOCAL = 'retry_local',      // Retry on same worker
  RETRY_DIFFERENT = 'retry_different', // Retry on different worker
  SKIP_TEST = 'skip_test',          // Skip this test

// Implementation with circuit breaker pattern
class CircuitBreaker {
  private failures: number = 0;
  private lastFailureTime: Date | null = null;
  private state: 'CLOSED' | 'OPEN' | 'HALF_OPEN' = 'CLOSED';
  
  constructor(
    private failureThreshold: number,
    private resetTimeoutMs: number
  ) {}
  
  async execute<T>(operation: () => Promise<T>): Promise<T> {
    if (this.state === 'OPEN') {
      if (this.isResetTimeoutExpired()) {
        this.state = 'HALF_OPEN';
      } else {
        throw new CircuitBreakerOpenError('Circuit breaker is open');
      }
    }
    
    try {
      const result = await operation();
      
      if (this.state === 'HALF_OPEN') {
        this.reset();
      }
      
      return result;
    } catch (error) {
      this.recordFailure();
      throw error;
    }
  }
  
  private recordFailure(): void {
    this.failures++;
    this.lastFailureTime = new Date();
    
    if (this.failures >= this.failureThreshold) {
      this.state = 'OPEN';
      setTimeout(() => {
        this.state = 'HALF_OPEN';
      }, this.resetTimeoutMs);
    }
  }
  
  private reset(): void {
    this.failures = 0;
    this.lastFailureTime = null;
    this.state = 'CLOSED';
  }
}

      spanId,

      logs: []
    };

    span.logs.push({
      timestamp: Date.now(),
      fields: {
        'error.object': error,
        'error.kind': error.name,
        'message': error.message,
        'stack': error.stack
      }
    });

    this.reportErrorToCentral(span, error);
  }

class ValidationResult {

    this.valid = valid;
    this.errors = errors;

  static valid() {
    return new ValidationResult(true);
  }
