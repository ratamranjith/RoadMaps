1.1: What is Software Testing?

Software testing is the process of evaluating a software application to verify that it functions correctly, meets requirements, and is free from defects. It's not just about finding bugs‚Äîit's about ensuring quality, reliability, and user satisfaction.

Key Insight: Testing is a quality assurance activity. Good testing helps catch defects early when they're cheaper and easier to fix, before they reach production.

  • üí∞ Cost Effective: Fixing bugs in production is 10x more expensive
  • ‚úÖ Quality Assurance: Ensures software meets specifications
  • üë• User Satisfaction: Delivers reliable, bug-free applications

  • üîí Security: Identifies vulnerabilities and security flaws

1.2: SDLC vs STLC

Software Development Life Cycle (SDLC)

SDLC is the overall process of developing software from conception to deployment and maintenance.

1. Planning
2. Analysis
3. Design
4. Development
5. Testing
6. Deployment
7. Maintenance

Software Testing Life Cycle (STLC)

STLC is a subset of SDLC, focusing specifically on the quality assurance and testing activities.

Aspect SDLC STLC
Scope Entire software development process Testing and QA activities only
Duration Months to years Part of SDLC, typically weeks
Phases Planning ‚Üí Design ‚Üí Development ‚Üí Testing ‚Üí Deployment Test Planning ‚Üí Design ‚Üí Execution ‚Üí Closure
Participants Everyone (managers, developers, testers, DevOps) QA team, developers, business analysts
Outcome Complete software product Quality assurance report, test cases, bug reports
Key Takeaway:
  • SDLC is the big picture (how software is built)
  • STLC is testing's role within SDLC (how software is verified)
  • Testing happens throughout SDLC, not just at the end

1.4: Bug Lifecycle

Understanding how bugs flow through the system helps with communication and prioritization.

NEW
ASSIGNED
IN PROGRESS
FIXED
VERIFIED
CLOSED

Bug Status Definitions

  • NEW: Bug just reported, not yet reviewed
  • ASSIGNED: Bug assigned to a developer
  • IN PROGRESS: Developer is working on the fix
  • FIXED: Developer says bug is fixed, awaiting verification
  • VERIFIED: Tester confirms the fix works
  • CLOSED: Bug is resolved and closed
  • REOPENED: (If fix doesn't work) Bug goes back to IN PROGRESS
  • WONT FIX: Decided not to fix for business reasons
  • DUPLICATE: Same bug already reported
Pro Tip: Always provide clear, reproducible steps when reporting a bug. This helps developers fix it faster.

1.5: Bug Severity vs Priority

Severity (Technical Impact)

Level Description Example
CRITICAL Application crashes or core feature broken Payment button doesn't work
MAJOR Important feature doesn't work Login takes 10 seconds
MINOR Minor issue, workaround exists Button text alignment off
TRIVIAL Cosmetic issue Typo in confirmation message

Priority (Business Impact)

Level Description When to Fix
P1 (URGENT) Must fix before release Immediately
P2 (HIGH) Should fix before release ASAP
P3 (MEDIUM) Can fix in next release Next sprint
P4 (LOW) Nice to have fix Whenever possible
Important: A bug can be HIGH severity (technically broken) but LOW priority (rarely used feature). Conversely, a TRIVIAL severity (cosmetic) can be HIGH priority (company logo looks wrong).
