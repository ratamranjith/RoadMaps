1.1: What is Software Testing?

Software testing is the process of evaluating a software application to verify that it functions correctly, meets requirements, and is free from defects. It's not just about finding bugs‚Äîit's about ensuring quality, reliability, and user satisfaction.

Key Insight: Testing is a quality assurance activity. Good testing helps catch defects early when they're cheaper and easier to fix, before they reach production.

Why is Testing Important?

  • üéØ Prevents Production Issues: Catch bugs before users encounter them
  • üí∞ Cost Effective: Fixing bugs in production is 10x more expensive
  • ‚úÖ Quality Assurance: Ensures software meets specifications
  • üë• User Satisfaction: Delivers reliable, bug-free applications
  • ‚öñÔ∏è Legal Compliance: Some industries require rigorous testing (banking, healthcare)
  • üîí Security: Identifies vulnerabilities and security flaws

1.2: SDLC vs STLC

Software Development Life Cycle (SDLC)

SDLC is the overall process of developing software from conception to deployment and maintenance.

1. Planning
2. Analysis
3. Design
4. Development
5. Testing
6. Deployment
7. Maintenance

Software Testing Life Cycle (STLC)

STLC is a subset of SDLC, focusing specifically on the quality assurance and testing activities.

Aspect SDLC STLC
Scope Entire software development process Testing and QA activities only
Duration Months to years Part of SDLC, typically weeks
Phases Planning ‚Üí Design ‚Üí Development ‚Üí Testing ‚Üí Deployment Test Planning ‚Üí Design ‚Üí Execution ‚Üí Closure
Participants Everyone (managers, developers, testers, DevOps) QA team, developers, business analysts
Outcome Complete software product Quality assurance report, test cases, bug reports
Key Takeaway:
  • SDLC is the big picture (how software is built)
  • STLC is testing's role within SDLC (how software is verified)
  • Testing happens throughout SDLC, not just at the end

Functional vs Non-Functional Testing

Aspect Functional Testing Non-Functional Testing
Focus What the system does How well the system performs
Based On Business requirements, specifications Performance criteria, user expectations
Objective Validate software actions/outputs Validate software behavior/attributes
Examples Login, Payment processing, Search Load time, Security, Usability, Compatibility

Key Insight: Most projects use a balanced mix of manual and automated testing. Manual testing is ideal for exploratory and usability testing, while automated testing excels at regression and performance testing.

1.4: Static vs Dynamic Testing

Static Testing (Verification)

Examining software work products (documentation, code) without executing them.

Dynamic Testing (Validation)

Executing software with test data to observe runtime behavior.

Aspect Static Testing Dynamic Testing
When Performed Early in SDLC (requirements, design, coding phases) Later in SDLC (after executable code is available)
What is Tested Documents, requirements, code, designs Running application, actual output
Techniques Reviews, walkthroughs, inspections, static analysis Unit, integration, system, acceptance testing
Defect Finding Finds defects early (cheaper to fix) Finds runtime, functional defects
Cost to Fix Defects Low (early in lifecycle) High (later in lifecycle)
Automation Static analysis tools (SonarQube, ESLint) Test automation tools (Selenium, JMeter)
Best Practice: Effective testing strategy includes both static and dynamic testing. Static testing prevents defects from being introduced, while dynamic testing uncovers defects in the running application.

1.5: Bug Lifecycle

Understanding how bugs flow through the system helps with communication and prioritization.

NEW
ASSIGNED
IN PROGRESS
FIXED
VERIFIED
CLOSED

Bug Status Definitions

  • NEW: Bug just reported, not yet reviewed
  • ASSIGNED: Bug assigned to a developer
  • IN PROGRESS: Developer is working on the fix
  • FIXED: Developer says bug is fixed, awaiting verification
  • VERIFIED: Tester confirms the fix works
  • CLOSED: Bug is resolved and closed
  • REOPENED: (If fix doesn't work) Bug goes back to IN PROGRESS
  • WONT FIX: Decided not to fix for business reasons
  • DUPLICATE: Same bug already reported
Pro Tip: Always provide clear, reproducible steps when reporting a bug. This helps developers fix it faster.

1.6: Bug Severity vs Priority

Severity (Technical Impact)

Level Description Example
CRITICAL Application crashes or core feature broken Payment button doesn't work
MAJOR Important feature doesn't work Login takes 10 seconds
MINOR Minor issue, workaround exists Button text alignment off
TRIVIAL Cosmetic issue Typo in confirmation message

Priority (Business Impact)

Level Description When to Fix
P1 (URGENT) Must fix before release Immediately
P2 (HIGH) Should fix before release ASAP
P3 (MEDIUM) Can fix in next release Next sprint
P4 (LOW) Nice to have fix Whenever possible
Important: A bug can be HIGH severity (technically broken) but LOW priority (rarely used feature). Conversely, a TRIVIAL severity (cosmetic) can be HIGH priority (company logo looks wrong).

Core Testing Principles

Principle Description Implication for Testers
Exhaustive testing is impossible Cannot test all combinations of inputs/preconditions Use risk-based & priority-based testing
Defect clustering Small number of modules contain most defects Focus testing on critical/complex modules
Pesticide paradox Repeating same tests finds fewer new defects Regularly review & update test cases
Testing shows presence of defects Testing can prove defects exist, not their absence Testing reduces probability of undiscovered defects
Absence-of-errors fallacy No defects ‚â† useful/usable product Focus on requirements & user needs, not just bug count

Remember: No single testing approach fits all projects. The best testing strategy depends on project context, risks, timeline, and resources.

Fundamental Beginner

1. What is software testing and why is it important?

Answer Structure: Definition + Importance + Business Impact

Answer: Software testing is the process of evaluating a software application to identify differences between expected and actual results. It's not just about finding bugs but ensuring quality, reliability, and that the software meets requirements.

Importance includes:

  • Cost Efficiency: Fixing bugs in production is 5-10x more expensive than in early stages
  • Customer Satisfaction: Quality software leads to happy users and better retention
  • Security: Identifies vulnerabilities before they're exploited
  • Risk Management: Reduces business risks associated with software failures
  • Compliance: Many industries (healthcare, finance) require rigorous testing for regulatory compliance
Follow-up question they might ask: "Can you give an example of how poor testing cost a company significantly?"

Example answer: "The Knight Capital Group lost $440 million in 45 minutes due to untested deployment of trading software. Proper regression testing could have prevented this."

Testing Types Intermediate

2. Explain the difference between verification and validation.

Answer: This is often summarized as "Are we building the product right?" vs "Are we building the right product?"

Aspect Verification Validation
Focus Process-oriented (static testing) Product-oriented (dynamic testing)
Question Are we building the product right? Are we building the right product?
When Throughout development (early and continuous) After verifiable product is available
Methods Reviews, walkthroughs, inspections Testing, UAT, beta testing
Examples Code review, requirement analysis Functional testing, usability testing
Defects Found Early-stage defects (cheaper to fix) Runtime/functional defects

Practical example: Verification is checking that a login screen has all required fields (email, password, submit button) per specifications. Validation is actually trying to log in with valid/invalid credentials to see if it works correctly.

Test Design Intermediate

3. What is equivalence partitioning and boundary value analysis? Give examples.

Answer: These are black-box test design techniques that help reduce the number of test cases while maintaining coverage.

Equivalence Partitioning: Divides input data into valid and invalid partitions where all values in a partition should behave the same way.

  • Example: A field accepts ages 18-60
    • Valid partition: 18-60 (test with one value, e.g., 30)
    • Invalid partitions: <18 (e.g., 17) and >60 (e.g., 61)

Boundary Value Analysis: Tests at the boundaries between partitions since errors often occur at edges.

  • Example: Same age field 18-60
    • Test values: 17, 18, 19 (around lower boundary)
    • Test values: 59, 60, 61 (around upper boundary)
Scenario for practice: "How would you test a password field that must be 8-16 characters with at least one number and one special character?"

Think about: Valid partitions (8-16 chars with requirements), invalid partitions (too short, too long, missing requirements), boundary values (7,8,9 and 15,16,17).

Bug Management Intermediate

4. What information should a good bug report contain?

Answer: A comprehensive bug report should contain enough information for developers to reproduce, understand, and fix the issue.

Essential elements:

  1. Title/Summary: Clear, concise description (e.g., "Login fails with valid credentials after 3 attempts")
  2. Environment: OS, browser, device, app version, any relevant configuration
  3. Steps to Reproduce: Numbered, detailed, exact steps
    • Good: "1. Navigate to https://example.com/login 2. Enter test@email.com in email field 3. Enter Password123! in password field 4. Click Login button"
    • Bad: "Login doesn't work"
  4. Expected vs Actual Results: Clearly state both
    • Expected: "User should be redirected to dashboard"
    • Actual: "Error message 'Invalid credentials' appears"
  5. Severity & Priority: Your assessment of impact
  6. Attachments: Screenshots, videos, logs, network traces
  7. Test Data: Specific data used if relevant
  8. Frequency: How often it occurs (always, intermittent, once)

Pro tip: Imagine you're explaining the issue to someone who can't see your screen. Be specific enough that they could reproduce it exactly.

Scenario-Based Advanced

5. How would you test an e-commerce "Add to Cart" functionality?

Answer: I'd approach this systematically, covering functional, non-functional, and edge cases.

Functional Testing:

  • Add single item to cart
  • Add multiple quantities of same item
  • Add different items to cart
  • Verify cart updates in real-time (without page refresh)
  • Check cart persists across sessions (if logged in)
  • Verify item details in cart (price, name, image, options)
  • Test from different pages (product page, search results, recommendations)

Boundary & Error Cases:

  • Add item with zero/negative quantity
  • Add more than available stock
  • Add item that becomes unavailable after adding to cart
  • Test with special characters in product names
  • Test during high traffic (concurrency issues)

Integration Testing:

  • Cart updates when inventory changes
  • Price changes reflect in cart
  • Promotions/discounts apply correctly
  • Cart syncs across devices for logged-in users

Non-Functional Testing:

  • Performance: How fast does cart update? Load with 100+ items
  • Usability: Is the cart icon intuitive? Clear item count?
  • Accessibility: Can screen readers interpret cart status?

My approach: I'd start with happy path scenarios, then boundary cases, then error conditions, ensuring all requirements are covered with traceability back to specifications.

Process Intermediate

6. What is the difference between retesting and regression testing?

Answer: These are often confused but serve different purposes in the testing cycle.

Aspect Retesting Regression Testing
Purpose Verify specific bug fixes work Ensure new changes don't break existing functionality
What's Tested Only the fixed defects Impacted areas and related functionality
Test Cases Specific to the bug Broad suite of existing tests
When After developer fixes a bug After any change (bug fix, enhancement, config change)
Priority High (must verify fix before release) High (ensures overall stability)
Example Bug #123: Login fails with special characters - retest login with those specific characters After fixing login bug, test registration, password reset, profile update to ensure nothing else broke

Practical scenario: When a payment processing bug is fixed, retesting verifies that specific payment works. Regression testing verifies the entire checkout process, cart, inventory updates, email notifications, etc., still work correctly.

Automation Advanced

7. When should you automate tests vs. when should you do manual testing?

Answer: The decision depends on multiple factors including ROI, stability, and test type.

Automate When:
  • Repetitive tests: Same tests run frequently (regression suites)
  • Data-driven tests: Testing with multiple data sets
  • Load/performance tests: Simulating thousands of users
  • Stable functionality: Features unlikely to change soon
  • Complex calculations: Where manual verification is error-prone
  • Cross-browser/device testing: Running same tests on multiple configurations
Manual Testing Preferred When:
  • Exploratory testing: Learning the application, ad-hoc testing
  • Usability testing: Subjective user experience evaluation
  • One-time tests: Not worth automation effort
  • Frequently changing features: High maintenance cost for automation
  • Complex user interactions: Hard to automate reliably
  • Early development stages: When UI/functionality is unstable

Decision Framework: Consider the ROI equation: Automation effort vs. Number of executions √ó Time saved per execution. Also consider maintenance cost and skill availability.

Example: A login functionality used by thousands daily should be automated. A new feature in beta with weekly UI changes should be tested manually initially.

Behavioral Intermediate

8. Describe a time you found a difficult bug. How did you approach it?

Use STAR Method: Situation ‚Üí Task ‚Üí Action ‚Üí Result

Sample Answer Structure:

Situation: "In my previous project, we had an intermittent issue where the application would crash randomly for about 5% of users, but we couldn't reproduce it consistently in our test environment."

Task: "My task was to identify the root cause of this intermittent crash that was causing customer complaints and negatively impacting user experience."

Action: "I took a systematic approach:

  1. Analyzed crash logs and identified a pattern - crashes seemed related to memory usage
  2. Set up detailed monitoring to track memory usage during specific workflows
  3. Created a test harness to simulate extended user sessions
  4. Isolated the issue to a specific module that had a memory leak when processing large datasets
  5. Collaborated with developers to add more detailed logging
  6. Reproduced the issue by creating a specific test dataset that triggered the leak

Result: "We identified that a caching mechanism wasn't clearing properly. The fix reduced crashes by 95% and improved application stability. I also documented the debugging approach for similar future issues."

What interviewers look for: Problem-solving approach, persistence, technical skills, collaboration, and documentation.
