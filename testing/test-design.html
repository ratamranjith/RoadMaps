2.1: Test Planning Essentials

A test plan is a comprehensive document that outlines the entire testing approach for a project. It sets expectations and provides guidance for the testing team.

Why is Test Planning Important?

  • ‚úì Clarity: Everyone knows what will be tested and how
  • ‚úì Resource Management: Allocates time, people, and tools efficiently
  • ‚úì Risk Management: Identifies high-risk areas for focused testing
  • ‚úì Communication: Sets expectations with stakeholders
  • ‚úì Measurable Goals: Tracks progress against objectives

Test Plan Template

1. INTRODUCTION & OBJECTIVES - Project overview - Testing goals and objectives 2. SCOPE - What's included (features, modules) - What's excluded (out of scope items) 3. TESTING APPROACH - Testing strategy - Testing types (unit, integration, system, UAT) - Testing techniques (black box, white box) 4. RESOURCES & SCHEDULE - Team members and roles - Test environment details - Timeline and milestones 5. TEST DELIVERABLES - Test cases - Test execution reports - Bug reports - Final test summary 6. ENTRY & EXIT CRITERIA - When to start testing - When testing is complete 7. RISK ASSESSMENT - Potential risks and mitigation strategies
Pro Tip: A good test plan is a living document. Update it as requirements change and lessons are learned.

2.2: Test Design Techniques

Test design techniques help you create effective test cases that cover all important scenarios with minimal redundancy.

1. Equivalence Partitioning (EP)

Divide inputs into categories where all values should behave the same way.

Example - Age Field (18+):
Invalid: age < 0 ‚Üí Test: -5
Valid: age >= 18 ‚Üí Test: 25
Invalid: age < 18 ‚Üí Test: 15
Each partition gets ONE test (not all values)

2. Boundary Value Analysis (BVA)

Test the boundaries between equivalence partitions (where bugs often hide).

Example - Password (8-16 characters):
Test: 7 chars (just below)
Test: 8 chars (exactly at boundary)
Test: 16 chars (exactly at boundary)
Test: 17 chars (just above)
Boundary bugs are most common!

3. Decision Table Testing

Test all combinations of inputs and conditions systematically.

Test User Active? Has Permission? Expected Result
1 Yes Yes Access Granted
2 Yes No Access Denied
3 No Yes Account Inactive
4 No No Account Inactive

4. State Transition Testing

Test different states and transitions between them.

Example - Order States:
Pending ‚Üí Processing ‚Üí Shipped ‚Üí Delivered
Test each valid transition
Test invalid transitions (e.g., Shipped ‚Üí Pending should fail)
Which technique to use?
  • EP + BVA for input validation
  • Decision Tables for complex logic
  • State Transition for workflows
  • Combine multiple techniques for comprehensive coverage

2.3: Writing Effective Test Cases

A good test case is clear, concise, and can be executed by anyone on the team.

Test Case Template

TEST CASE ID: TC_LOGIN_001 TITLE: Verify successful login with valid credentials PRIORITY: P1 (High) SEVERITY: Critical PRECONDITIONS: - Application is running - User account exists in database - User is on login page TEST DATA: - Email: testuser@example.com - Password: ValidPass123 STEPS: 1. Enter valid email in Email field 2. Enter valid password in Password field 3. Click "Login" button 4. Observe dashboard loads EXPECTED RESULT: - Login button becomes disabled (shows loading) - Redirected to /dashboard within 2 seconds - Success message "Welcome, Test User" appears - Session token stored in browser ACTUAL RESULT: [To be filled during execution] STATUS: ‚òê PASS ‚òê FAIL ‚òê SKIP NOTES: - Tested on Chrome 120, Windows 11 - No console errors observed

Test Case Best Practices

  • ‚úì One assertion per test case - Each test should verify one specific behavior
  • ‚úì Unique ID - Use format: TC_[FEATURE]_[NUMBER]
  • ‚úì Clear steps - A non-technical person should understand
  • ‚úì Specific expected results - Not "it should work" but "message appears in 2 seconds"
  • ‚úì Test data included - Don't assume what data to use
  • ‚úì Preconditions noted - What setup is required
  • ‚úì Reusable - Can be run multiple times with same expected results

2.4: Risk-Based Testing

When time is limited, focus testing on high-risk areas using a risk matrix.

Risk Assessment Matrix

Risk Level Probability Impact Testing Effort
CRITICAL High High Maximum (80%)
HIGH High Medium High (15%)
MEDIUM Medium Medium Moderate (4%)
LOW Low Low Minimal (1%)

High-Risk Areas to Focus On

  • üí∞ Payment/Billing: Financial transactions are critical
  • üîê Authentication/Security: Users' data must be protected
  • üíæ Data Persistence: Data must be saved and retrieved correctly
  • üîÑ Integrations: Third-party APIs are common failure points
  • ‚ö° Performance: Slow app = bad user experience
  • üì± Cross-browser: Compatibility issues affect many users

2.5: Traceability Matrix

Ensures all requirements have corresponding test cases and that tests cover all requirements.

Requirement ID Requirement Description Test Case ID Status
REQ_001 User must login with email & password TC_LOGIN_001, TC_LOGIN_002 ‚úì Covered
REQ_002 Invalid credentials should show error TC_LOGIN_003, TC_LOGIN_004 ‚úì Covered
REQ_003 Password must be 8+ characters TC_LOGIN_005 ‚úì Covered
REQ_004 Account lockout after 5 failed attempts ‚úó NOT FOUND ‚ùå Not Covered
Benefit: Traceability matrix prevents missed requirements and identifies redundant tests.
